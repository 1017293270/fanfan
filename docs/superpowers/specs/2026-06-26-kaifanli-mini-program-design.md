# 开饭狸微信小程序 MVP 设计规格

日期：2026-06-26  
状态：已确认设计，待用户复核后进入实现计划

## 1. 项目概述

`开饭狸` 是一个解决“今天吃什么”的微信小程序。用户打开小程序后，通过当前位置、一句话偏好和少量快选条件，让系统从附近餐厅中给出 1 个主推荐和 2 个备选，并提供换一批、收藏、不喜欢和导航入口。

产品核心不是做大而全的餐厅列表，而是减少选择压力：用户说出“今天想怎么吃”，`饭饭狸` 帮用户快速拍板。

## 2. 已确认决策

- 小程序形态：微信原生小程序。
- 推荐体验：默认给 1 个主推荐 + 2 个备选，允许换一批。
- 偏好输入：一句话自然语言 + 快选偏好芯片。
- 餐厅数据：真实接入高德 API，使用周边搜索 + POI 详情补充。
- 后端形态：独立后端服务。
- AI 形态：OpenAI-compatible Adapter，通过环境变量配置模型、Base URL 和 Key。
- AI 职责：解析自然语言偏好，并生成推荐理由；餐厅筛选和排序由后端规则控制。
- 账号体系：MVP 不强制登录，本地保存最近偏好、收藏和不喜欢的店铺。
- 品牌名：开饭狸。
- 角色名：饭饭狸。
- 小程序头像：`assets/brand/kaifanli-avatar-144.png`。

## 3. MVP 目标

### 3.1 用户目标

- 快速表达当前想吃什么。
- 基于当前位置看到附近真实可去的餐厅。
- 直接获得一个明确推荐，不再继续面对长列表。
- 推荐理由简短、有用、像一个饭搭子给建议。
- 对不喜欢的店可以排除，对喜欢的店可以收藏。

### 3.2 产品目标

- 首次使用无需登录即可完成推荐。
- 单次推荐流程控制在 30 秒内。
- 推荐结果稳定可解释，避免完全由 AI 随机拍脑袋。
- 后端接口和 AI/高德配置可替换，便于后续扩展。
- 视觉上形成可识别的品牌角色和小程序气质。

## 4. 非目标

MVP 不做以下内容：

- 强制微信登录和云端用户画像。
- 多人饭局投票。
- 商家后台。
- 真实排队数据。
- 复杂路线规划和实时步行时间。
- 复杂评分体系或点评内容抓取。
- 支付、团购、优惠券。
- 完整内容社区。

## 5. 品牌套件

### 5.1 品牌定位

`开饭狸` 是一个亲切、轻松、能替用户拍板的吃饭决策助手。它应该像一个每天中午都在旁边帮忙拿主意的饭搭子，而不是严肃的搜索工具。

### 5.2 角色系统

主角色 `饭饭狸` 是一只小熊猫吃饭助手。它的作用是降低决策压力，而不是只做装饰。

角色出现位置：

- App 头像和品牌入口。
- 首页问候区。
- 推荐理由标题。
- 加载状态。
- 空状态。
- 收藏成功、换一批、不喜欢等轻反馈。

角色语气：

- 亲切、直接、少废话。
- 可以可爱，但不幼稚。
- 多用“帮你拍板”“就吃它”“附近好吃的正在赶来”这类短句。

### 5.3 视觉风格

整体风格是“软萌实用”。

主色：

- 番茄红：品牌识别和主行动按钮。
- 米白：页面背景和卡片底色。
- 翠竹绿：位置、营业中、健康/清淡等辅助信息。
- 暖橙黄：食欲、强调、推荐亮点。
- 深墨色：正文文字和高对比信息。
- 浅桃色：弱提示和状态背景。

视觉约束：

- 避免全页面都是单一红色。
- 避免过度奶油风、儿童风和复杂阴影。
- 卡片圆角柔和，但信息层级要清楚。
- 按钮、芯片和卡片尺寸稳定，避免内容变化导致布局跳动。

### 5.4 图标语言

图标尽量和饭饭狸或吃饭场景相关：

- 换一批：爪印刷新。
- 收藏：爪心爱心。
- 不喜欢：小熊猫皱眉或爪印叉。
- 附近：竹叶定位。
- 导航：筷子箭头。
- 预算：小碗金币。
- 辣度：辣椒小脸。
- 就吃它：小碗闪光。

MVP 阶段图标可以先使用 PNG/WebP 资产，后续再按需要矢量化。

### 5.5 文案规范

核心文案：

- 小程序介绍：`开饭狸根据你的位置、口味和预算，从附近餐厅里挑出今日推荐，少纠结，快开饭。`
- 品牌语：`今天吃什么，让饭饭狸来拍板。`
- 首页提问：`今天想吃什么？`
- 输入占位：`比如：想吃热乎的，不太辣，别太远`
- 推荐标题：`饭饭狸推荐`
- 主行动：`就吃它`
- 导航行动：`带我去`
- 换推荐：`换一批`
- 排除店铺：`这家先不看`
- 空状态：`附近没找到合适的，换个条件试试。`
- 加载状态：`附近好吃的正在赶来。`

## 6. 用户流程

1. 用户进入首页。
2. 小程序请求定位授权。
3. 用户输入一句话偏好，或直接使用默认偏好。
4. 用户选择快选条件：距离、预算、辣度、想吃类型、是否只看营业中。
5. 小程序提交位置和偏好到后端。
6. 后端调用 AI 解析偏好。
7. 后端调用高德周边搜索获取附近餐饮 POI。
8. 后端对候选 POI 补充详情。
9. 后端根据结构化偏好、距离、价格、标签、评分、营业状态和本地排除列表进行排序。
10. 后端选出 1 个主推荐和 2 个备选。
11. 后端调用 AI 生成简短推荐理由。
12. 小程序展示推荐结果。
13. 用户选择就吃它、换一批、收藏或不喜欢。

## 7. 页面与组件

### 7.1 首页

首页是 MVP 的主界面，承担偏好输入和推荐展示。

页面区域：

- 顶部位置栏：显示当前位置或“重新定位”。
- 品牌问候区：显示饭饭狸头像和一句引导。
- 一句话输入框：支持用户输入自然语言偏好。
- 快选偏好芯片：距离、预算、辣度、想吃类型、营业中。
- 推荐结果区：展示主推荐和两个备选。
- 底部行动栏：换一批、收藏、不喜欢、就吃它/带我去。

### 7.2 收藏与本地记录

MVP 可以先做轻量收藏入口，不强制做完整个人中心。

本地保存：

- 最近一次偏好。
- 收藏 POI 列表。
- 不喜欢 POI 列表。
- 最近推荐结果摘要。

数据保存在微信小程序本地存储中。用户清缓存后数据可以丢失，这是 MVP 可接受限制。

### 7.3 状态设计

- 初始状态：展示默认偏好芯片和输入提示。
- 定位未授权：解释为什么需要定位，并提供手动重试。
- 加载状态：饭饭狸加载图 + `附近好吃的正在赶来。`
- 推荐成功：主推荐卡 + 备选列表。
- 无结果：建议放宽距离、预算、辣度或餐饮类型。
- 接口错误：提示稍后重试，保留用户输入。
- AI 解析失败：降级到快选条件和关键词规则。

## 8. 推荐逻辑

### 8.1 偏好解析

AI 将用户自然语言解析为结构化偏好：

```json
{
  "distanceMeters": 1500,
  "budgetPerPerson": 30,
  "spicyPreference": "mild",
  "cravings": ["热乎", "米饭"],
  "avoid": ["太辣", "太远"],
  "mealScene": "工作日午餐",
  "mustOpenNow": true
}
```

解析失败时，后端使用快选条件和关键词规则兜底。

### 8.2 POI 获取

高德 API 调用策略：

- 周边搜索：以当前位置、半径和餐饮关键词获取 POI 列表。
- 详情补充：对候选池前若干项补充详情字段。
- 候选数量：MVP 可先取 20-30 个 POI，再筛选 3 个结果。

高德 Key 只存在后端环境变量中，小程序端不保存密钥。

### 8.3 排序规则

排序由后端规则控制，AI 不直接决定最终排序。

基础评分维度：

- 距离匹配度。
- 预算匹配度。
- 口味/菜系匹配度。
- 是否营业中。
- 评分或热度字段。
- 用户本地不喜欢列表排除。
- 收藏店铺可适当加权，但不强制重复推荐。

输出结构：

- `primaryRecommendation`：主推荐。
- `alternatives`：两个备选。
- `reason`：饭饭狸推荐理由。
- `debugSignals`：开发环境可返回排序依据，生产环境默认不展示。

## 9. 技术架构

### 9.1 总体架构

```text
微信原生小程序
  -> 独立后端 API
    -> AI Adapter（OpenAI-compatible）
    -> 高德 Web 服务 API
    -> 推荐排序模块
```

### 9.2 小程序端

职责：

- 定位授权和当前位置获取。
- 偏好输入和快选条件管理。
- 本地收藏、不喜欢、最近偏好存储。
- 调用后端推荐接口。
- 展示推荐结果和状态反馈。

建议目录边界：

- `pages/index`：首页和推荐结果。
- `pages/favorites`：收藏列表，MVP 可延后但接口预留。
- `components/preference-chip`：偏好芯片。
- `components/recommendation-card`：主推荐卡。
- `components/restaurant-row`：备选餐厅行。
- `components/mascot-state`：加载、空状态、错误状态。
- `utils/storage`：本地存储封装。
- `utils/api`：后端请求封装。

### 9.3 独立后端

建议技术：Node.js + Fastify。

选择原因：

- 接口轻量，适合 MVP。
- Schema 校验清晰。
- 后续可扩展日志、限流、缓存和任务队列。

核心模块：

- `config`：读取环境变量。
- `routes/recommendations`：推荐接口。
- `services/amap`：高德 API 封装。
- `services/ai`：OpenAI-compatible Adapter。
- `services/preference-parser`：偏好解析。
- `services/ranker`：候选餐厅排序。
- `services/reason-writer`：推荐理由生成。
- `schemas`：请求和响应结构。

## 10. 配置与安全

环境变量：

- `AMAP_API_KEY`
- `OPENAI_COMPATIBLE_API_KEY`
- `OPENAI_COMPATIBLE_BASE_URL`
- `OPENAI_COMPATIBLE_MODEL`
- `API_PORT`
- `NODE_ENV`

安全约束：

- AI Key 和高德 Key 不进入小程序端。
- 小程序只请求独立后端合法域名。
- 后端需要基础限流，防止接口被刷。
- 日志不记录完整密钥，不记录用户精确长期轨迹。
- 推荐请求可记录 requestId 和错误类型，便于排查。

## 11. 错误处理与降级

- 定位失败：提示用户重新授权或手动选择附近范围。
- 高德搜索失败：提示网络异常，并允许重试。
- POI 详情失败：保留基础搜索结果继续排序。
- AI 偏好解析失败：使用快选条件和关键词规则。
- AI 理由生成失败：使用模板理由，例如 `这家距离和口味都比较符合你刚才的选择。`
- 无合适结果：建议用户扩大距离、放宽预算或换一个想吃类型。
- 后端超时：小程序显示重试，不清空输入。

## 12. 测试策略

### 12.1 后端测试

- 偏好解析结果 schema 校验。
- 推荐排序规则单元测试。
- 高德服务封装的错误和空结果处理。
- AI Adapter 的超时、错误和降级测试。
- `/api/recommendations` 请求/响应契约测试。

### 12.2 小程序测试

- 首次定位授权流程。
- 定位拒绝后的提示和重试。
- 输入偏好 + 快选条件提交。
- 推荐成功、加载、无结果、错误状态。
- 收藏、不喜欢、换一批的本地存储行为。
- 头像和关键 UI 在小屏幕上不遮挡正文。

### 12.3 设计 QA

- 主推荐卡信息是否 3 秒内能读懂。
- 饭饭狸角色是否增强信任，而不是抢注意力。
- 按钮文案是否明确下一步动作。
- 小程序头像在 144x144 下是否清晰。
- 页面配色是否保持食欲感和可读性。

## 13. MVP 验收标准

- 用户可以授权定位并发起一次真实附近餐厅推荐。
- 用户可以输入自然语言偏好，并看到结构化推荐结果。
- 系统展示 1 个主推荐和 2 个备选。
- 推荐理由与用户偏好和餐厅信息相关。
- 用户可以换一批、收藏、不喜欢和打开导航入口。
- 本地保存最近偏好、收藏和不喜欢列表。
- 高德 Key 和 AI Key 只在后端配置。
- 小程序头像、品牌名称、简介和核心视觉一致。
